## Summary

Ran `queries/whoami.spl` (`index=sysmon whoami`) against Splunk over the last 24 hours against synthetic Sysmon-style test data generated for this exercise. The search returned **5 events**. Four are consistent with routine administrative use of `whoami` ([[T1033]] — System Owner/User Discovery), run by interactive/service accounts under expected parent processes (`cmd.exe`, `powershell_ise.exe`, `taskeng.exe`).

The fifth event stands out: on `FINANCE-PC03.corp.local`, user `CORP\jdoe` had `cmd.exe` spawned directly by `WINWORD.EXE` (Microsoft Word), which then ran `whoami /all` and redirected output to a file under `%TEMP%`. Word's `ParentCommandLine` shows it had opened `Invoice_2026_Q3.docm` — a macro-enabled document. Office applications spawning a command interpreter is a well-known indicator of malicious macro execution ([[T1204.002]] — User Execution: Malicious File) followed by an attacker/macro running a discovery command via the command shell ([[T1059.003]] — Command and Scripting Interpreter: Windows Command Shell) to enumerate the current user's privileges ([[T1033]]). This lineage (Office app → shell → discovery command, with output captured to a temp file) is consistent with early post-compromise reconnaissance following a phishing document, though no further activity (e.g. C2 callback, lateral movement) is present in this result set to confirm follow-on impact.
